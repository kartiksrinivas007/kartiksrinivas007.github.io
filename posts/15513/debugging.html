Debugging Tips and Tricks

  • Use Valgrind
  • Use GDB
  • Make assertion statements where you can, always helpful
  • check all branches of the control flow, always
  • Update the condition, see what happens, do it piece by piece

Good code is performance, available, portable, scalable, secure, testable and usable and READABLE. Do things algorithmically, split the task into multiple components, each of which can be testable

  • Adding tiny tests as you go makes it much easier to verify things are working
  • Before that – test your parsing by outputting our logging format

Think separation of concerns, modularity and reusability, DRY and much more Each module is responsible for its own internals No outside code “intrudes” on the inner workings of another module Use dictionary words to make sure that the code is well done use the four word maximum rule, limit names to the number of words that people can read at a glance This resource is gold

There is also a gdb cheatsheet on the website for the intro to computer systems book usee bt for frame up and down for moving up frames etc list x for 10 lines of code from line x optionally pass the function name also, use finish to finish the present function

use watch <variable>/value to see where that value or variable changes use x to examin the memory location and see what is there are that place (useful for mallocs surely)

use call expr to evaluate the function without the void return values, or use set [variable] expression to change the value assosciated with a varble or a memory address
